# Platform Architecture

## Design principle

The platform separates evidence, controls, and explanation:

1. BigQuery returns the analytical evidence.
2. Certified definitions establish the meaning and grain of metrics.
3. Deterministic checks establish source health and trust signals.
4. SQL parsing and governance rules detect violations.
5. The LLM explains only the structured facts produced by the preceding layers.

## Shared dataset

Demo dataset: `bigquery-public-data.thelook_ecommerce`

Core tables:

- `users`
- `products`
- `orders`
- `order_items`
- `inventory_items`
- `distribution_centers`
- `events`

## Repository layout

The platform is a monorepo with one shared BigQuery gateway and three independent Streamlit apps:

```text
shared/                       One BigQuery gateway (data_service, metric_catalog,
                               incidents, trust_scoring, audit, models) used by all apps.
apps/
  loupe_agent/                 Assistant layer.
    metrics.py                 Agent-facing and dashboard query functions.
    scenarios.py                Lever-based scenario baselines.
    validation.py                Closed-allowlist validation of LLM-extracted values.
    source_health.py              Per-table source-health lookup and honesty guarantee.
    chat.py                    The only module that talks to the LLM.
    ui.py                      Streamlit rendering; calls the modules above.
    main.py                    Thin entry point.
  metric_governance/          Definition layer (review.py, definition_diff.py,
                               remediation.py, explanations.py, chat.py, ui.py, main.py).
  data_quality_triage/        Reliability layer (checks.py, anomaly_engine.py,
                               profiling.py, incident_lifecycle.py, remediation.py,
                               explanations.py, chat.py, ui.py, main.py).
tests/                        Mirrors apps/ and shared/, one test package per module.
```

Each app is framework-independent of the others: none of the three imports from another app's `apps/<name>/` package. The only code shared across all three is `shared/`.

## Component boundaries

### Data service

`shared/data_service.py` is the **only** module in the monorepo allowed to construct a `bigquery.Client` or execute SQL/metadata calls. It owns parameterized query execution (`run_query()` -- sqlglot-enforced read-only SQL, named-parameter binding, byte-billed ceiling, timeout), metric catalog access, audit writes, incident reads and writes, and source-health lookup (`derive_source_health()`).

Authentication uses Application Default Credentials via `get_bigquery_client()`; no service-account JSON keys are created or read by any app.

`derive_source_health()` depends on a `loupe_platform.incidents` table that does not exist yet in this environment -- that table is Phase 6 (persistence) work, not yet started. Until it exists, calling it raises, and any app that needs source health must treat that as an explicit "unknown" status rather than assuming health. See `apps/loupe_agent/source_health.py` for the current pattern: every exception from `derive_source_health()` (including "table not found") maps to `{"status": "unknown", "known": False}`, which is treated as warning-worthy -- never silently upgraded to "healthy" and never given trust benefit.

### Metric catalog

`shared/metric_catalog.py` defines the certified metric catalog (`MetricDefinition` entries for revenue, margin, return_rate, margin_leakage, channel_mix). As of this writing every entry's `certification_status` is `"pending_validation"`. No app -- including narration text generated by an LLM -- may describe these as certified; each app-facing response that uses one of these metrics carries and displays its actual `certification_status`. See `apps/loupe_agent/chat.py`'s `certification_note()` for the pattern that grounds LLM narration in this status on every call.

Each definition's `measurement_grain` states the atomic business entity the metric is defined over (all five are `order_item`-grain) -- it is never a claim about any one query's output shape. That per-query shape (day/month/category/state/whole-window) is reporting grain, declared separately by each query function and by the UI/narration layer that renders its result. See docs/contracts.md's "Measurement grain vs. reporting grain" section for the full distinction and why conflating the two was a real bug this platform already made once.

### Review engine

`apps/metric_governance/review.py` parses SQL with `sqlglot` and returns structured findings. It should not render UI or rely on prose generated by an LLM to decide whether SQL is compliant.

### Reliability engine

`apps/data_quality_triage/checks.py`, `anomaly_engine.py`, and `incident_lifecycle.py` execute deterministic checks, calculate trust scores (via `shared/trust_scoring.py`), classify severity, and create structured incident data. LLM-generated playbooks (`explanations.py`) are downstream of these decisions.

### Assistant / query layer

`apps/loupe_agent/metrics.py` is the only module in the Assistant layer that calls `shared.data_service.run_query()`. `apps/loupe_agent/chat.py` classifies a question into one of a fixed set of intents via a closed allowlist dispatch table (`INTENT_HANDLERS`), extracts bounded parameters via the LLM, validates those parameters against closed sets in `apps/loupe_agent/validation.py` (rejecting anything outside the allowlist rather than attempting to sanitize it), and only then calls exactly one registered `metrics.py`/`scenarios.py` function with the validated value. The LLM has no path to generate warehouse SQL, select arbitrary tables or columns, or execute a query outside this registered tool set.

### User interface

Each app's `ui.py` renders application state and calls that app's own service functions. It must not contain direct warehouse queries or duplicate certified business logic.

## Cross-layer flow

```text
Business question or SQL
          |
          v
Metric definitions + SQL review + source health
          |
          v
Parameterized BigQuery query or deterministic check
          |
          v
Structured evidence, findings, and trust score
          |
          v
Plain-English explanation and recommended action
          |
          v
Audit event
```
